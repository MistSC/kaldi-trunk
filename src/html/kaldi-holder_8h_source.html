32 // The Table class uses a Holder class to wrap objects, and make them behave
33 // in a "normalized" way w.r.t. reading and writing, so the Table class can
34 // be template-ized without too much trouble. Look below this
35 // comment (search for GenericHolder) to see what it looks like.
36 //
37 // Requirements of the holder class:
38 //
39 // They can only contain objects that can be read/written without external
40 // information; other objects cannot be stored in this type of archive.
41 //
42 // In terms of what functions it should have, see GenericHolder below.
43 // It is just for documentation.
44 //
45 // (1) Requirements of the Read and Write functions
46 //
47 // The Read and Write functions should have the property that in a longer
48 // file, if the Read function is started from where the Write function started
49 // writing, it should go to where the Write function stopped writing, in either
50 // text or binary mode (but it's OK if it doesn't eat up trailing space).
51 //
52 // [Desirable property: when writing in text mode the output should contain
53 // exactly one newline, at the end of the output; this makes it easier to
54 // manipulate]
55 //
56 // [Desirable property for classes: the output should just be a binary-mode
57 // header (if in binary mode and it's a Kaldi object, or no header
58 // othewise), and then the output of Object.Write(). This means that when
59 // written to individual files with the scp: type of wspecifier, we can
60 // read the individual files in the "normal" Kaldi way by reading the
61 // binary header and then the object.]
62 //
63 //
64 // The Write function takes a 'binary' argument. In general, each object will
65 // have two formats: text and binary. However, it's permitted to throw() if
66 // asked to read in the text format if there is none. The file will be open, if
67 // the file system has binary/text modes, in the corresponding mode. However,
68 // the object should have a file-mode in which it can read either text or binary
69 // output. It announces this via the static IsReadInBinary() function. This
70 // will generally be the binary mode and it means that where necessary, in text
71 // formats, we must ignore \r characters.
72 //
73 // Memory requirements: if it allocates memory, the destructor should
74 // free that memory. Copying and assignment of Holder objects may be
75 // disallowed as the Table code never does this.
