40 // The Output and Input classes handle stream-opening for "extended" filenames
41 // that include actual files, standard-input/standard-output, pipes, and
42 // offsets into actual files. They also handle reading and writing the
43 // binary-mode headers for Kaldi files, where applicable. The classes have
44 // versions of the Open routines that throw and do not throw, depending whether
45 // the calling code wants to catch the errors or not; there are also versions
46 // that write (or do not write) the Kaldi binary-mode header that says if it's
47 // binary mode. Generally files that contain Kaldi objects will have the header
48 // on, so we know upon reading them whether they have the header. So you would
49 // use the OpenWithHeader routines for these (or the constructor); but other
50 // types of objects (e.g. FSTs) would have files without a header so you would
51 // use OpenNoHeader.
52 
53 // We now document the types of extended filenames that we use.
54 //
55 // A "wxfilename" is an extended filename for writing. It can take three forms:
56 // (1) Filename: e.g. "/some/filename", "./a/b/c", "c:\Users\dpovey\My
57 // Documents\\boo"
58 // (whatever the actual file-system interprets)
59 // (2) Standard output: "" or "-"
60 // (3) A pipe: e.g. "gunzip -c /tmp/abc.gz |"
61 //
62 //
63 // A "rxfilename" is an extended filename for reading. It can take four forms:
64 // (1) An actual filename, whatever the file-system can read, e.g. "/my/file".
65 // (2) Standard input: "" or "-"
66 // (3) A pipe: e.g. "| gzip -c > /tmp/abc.gz"
67 // (4) An offset into a file, e.g.: "/mnt/blah/data/1.ark:24871"
68 // [these are created by the Table and TableWriter classes; I may also write
69 // a program that creates them for arbitrary files]
70 //
71

74 // ...

80 // Output ko(some_filename, binary);
81 // MyObject1.Write(ko.Stream(), binary);
82 // MyObject2.Write(ko.Stream(), binary);

180 // Input interprets four kinds of filenames:
181 // (1) Normal filenames
182 // (2) The empty string or "-", interpreted as standard output
183 // (3) Pipes, e.g. "| gzip -c > some_file.gz"

198  // Open opens the stream for reading (the mode, where relevant, is binary; use
199  // OpenTextMode for text-mode, we made this a separate function rather than a
200  // boolean argument, to avoid confusion with Kaldi's text/binary distinction,
201  // since reading in the file system's text mode is unusual.) If
202  // contents_binary != NULL, it reads the binary-mode header and puts it in the
203  // "binary" variable. Returns true on success. If it returns false it will
204  // not be open. You may call Open even if it is already open; it will close
205  // the existing stream and reopen (however if closing the old stream failed it
206  // will throw).
207  inline bool Open(const std::string &rxfilename, bool *contents_binary = NULL);
208 
209  // As Open but (if the file system has text/binary modes) opens in text mode;
210  // you shouldn't ever have to use this as in Kaldi we read even text files in
211  // binary mode (and ignore the \r).
212  inline bool OpenTextMode(const std::string &rxfilename);
213 
214  // Return true if currently open for reading and Stream() will
215  // succeed. Does not guarantee that the stream is good.
216  inline bool IsOpen();
217 
218  // It is never necessary or helpful to call Close, except if
219  // you are concerned about to many filehandles being open.
220  // Close does not throw.
221  void Close();
